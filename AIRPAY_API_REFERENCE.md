# Airpay API — локальная рабочая справка

Источник: закрытая документация Airpay, просмотрена 23 июля 2026 года.

Эта справка сохраняет контракт и особенности интеграции в удобном для проекта виде. Она не содержит логинов, паролей, API-ключей, адресов стендов или данных плательщиков. При изменении контракта первоисточником остаётся документация Airpay.

## Доступ и транспорт

- Airpay выдаёт адреса тестового и боевого стендов отдельно после согласования и сетевой анкеты.
- Базовый путь API: `<base-url>/api/v20/`.
- Все методы: `POST`, HTTPS, JSON в UTF-8.
- Заголовки каждого запроса:

  ```http
  Authorization: Basic <base64(login:password)>
  Content-Type: application/json
  Accept: application/json
  ```

- Логин и пароль выдаёт процессинг. Их нужно хранить только в серверных переменных окружения; браузер и клиентский код не должны получать эти значения.

## Методы

| Метод | Путь | Назначение |
| --- | --- | --- |
| Каталог услуг | `/services` | Доступные услуги, поля формы и правила их проверки. |
| Описание услуги | `/service` | Карточка одной услуги по `serviceId`. |
| Баланс агента | `/balance` | Баланс, пороги отключения и валюта договора. |
| Проверка | `/check` | Проверяет реквизиты/сумму и получает данные для клиента. |
| Проведение платежа | `/pay` | Инициирует оплату после успешной проверки. |

В просмотренной документации нет отдельного endpoint для статуса платежа и нет описания вебхуков. Для незавершённой операции нужно повторять тот же запрос с тем же идентификатором.

## Каталог услуг

`POST /services` не требует тела. В ответе приходит `services[]`:

- `serviceId`, `name`, `type` — код, название и схема услуги;
- `group`, `country`, `fixedPayment` — метаданные каталога;
- `inputs[]` — поля, которые вводит оператор/клиент: `name`, `required`, `title`, `regexp`;
- `displays[]` — поля, которые Airpay может вернуть после `check` для показа клиенту; у них такой же контракт.

Перед `check` и `pay` нужно проверять введённые значения по `inputs[].regexp`. Если требуется описание одной услуги, `POST /service` получает тело:

```json
{ "serviceId": "A0001" }
```

## Схемы оплаты

1. **Простая.** Пользователь вводит одно или несколько полей; после `check` можно оплатить сразу либо сначала показать `displays`.
2. **С выбором.** `check` возвращает `contracts[]` — счета, договоры или заказы. Пользователь выбирает нужную сущность перед оплатой.
3. **Комплексная.** `check` может вернуть `invoice.invoices[]` с несколькими услугами, суммами, показаниями счётчиков и другими данными квитанции.

Форма и следующий экран должны строиться из ответа Airpay, а не из жёстко заданного списка полей.

## Общие поля платежа

| Поле | Назначение |
| --- | --- |
| `serviceId` | Код услуги Airpay. |
| `account` | Идентификатор плательщика: телефон, лицевой счёт, логин, email и т. п. |
| `agentTransactionId` | Уникальный номер операции в нашей системе. |
| `agentTransactionDate` | Время получения запроса от клиента: `YYYY-MM-DDThh:mm:ss`. |
| `amountTo` | Сумма зачисления; десятичное число с двумя знаками. |
| `amountFrom` | Сумма, принятая от клиента; опциональна. |
| `extras` | Дополнительные поля конкретной услуги. |
| `identityCard`, `fullName` | Опциональные идентификационные данные плательщика, если их требует услуга/законодательство. |

В `extras` могут передаваться `account1`, `account2` и иные поля из каталога. Платёжные суммы следует обрабатывать как десятичные значения, не как `float`.

## Проверка: `POST /check`

Минимальный запрос:

```json
{
  "serviceId": "A1011",
  "account": "7770017711",
  "agentTransactionId": 1233355,
  "agentTransactionDate": "2026-07-23T12:00:00",
  "amountTo": 200.00,
  "amountFrom": 200.00
}
```

Ответ всегда содержит `result`, `resultMessage`, `transactionId` и `agentTransactionId`. Дополнительно могут прийти:

- `displays` — текстовые и расчётные данные для подтверждения;
- `contracts[]` с `contractId`, названием, номером, суммой и датой;
- `invoice.invoices[]` с деталями квитанции;
- `currencyRate`, `rate`, `finalAmount`, `currency` — для международных провайдеров.

Нужно сохранить в транзакции исходный запрос, ответ проверки, `transactionId`, выбранный пользователем контракт/счёт и данные, необходимые для неизменного повторного запроса.

## Оплата: `POST /pay`

Базовый запрос повторяет платёжные поля из `check`; для `pay` обязательны `serviceId`, `account`, `agentTransactionId`, `agentTransactionDate`, `amountTo`. Состав тела для услуг с выбором или комплексных услуг определяется ответом `check` и тестовым каталогом Airpay.

Успешный ответ содержит `result: 0`, `transactionId`, `agentTransactionId`; также могут прийти `transactionDate` и `statusDate`.

## Статусы, идемпотентность и повторы

- `result: 0` — финальный успех.
- `result: 1`, `153`, `220`, `255` — временные состояния; повторять тот же запрос с увеличивающимся интервалом.
- Повтор не должен менять тело уже отправленного `pay`-запроса.
- В Airpay не должно быть двух успешно завершённых операций одного агента с одним `agentTransactionId`. Повтор с тем же номером возвращает результат прежней обработки.
- `result: 202` при первом запросе финальный. Если он пришёл при повторе операции, которая уже была во временном состоянии, исходная операция может продолжить обработку — её нельзя считать отменённой только по этому ответу.

Основные финальные ошибки: `4` — неверный формат идентификатора, `5` — плательщик не найден, `7` — приём отключён провайдером, `8` — временная техническая недоступность, `130` — услуга не реализована, `155` — запрещена, `215` — номер операции уже зарегистрирован, `241`/`242` — сумма вне допустимого диапазона, `300` — прочая ошибка.

В тестовом контуре также получен код `204` — «Приём платежа в указанной валюте невозможен». Этот код отсутствует в таблице документации; его нужно считать финальным отказом и выбирать услугу/валюту, доступную договору агента.

## Баланс: `POST /balance`

Тело не требуется. В ответе `agent` содержит `agentId`, `agentName`, `balance`, `overdraft`, необязательный `notification` и `currency`. Баланс следует показывать владельцу и использовать для предупреждения до остановки приёма платежей.

## Рекомендуемая реализация в Gamesales

1. Добавить отдельный серверный адаптер Airpay, не смешивая его с InterHub.
2. Хранить `AIRPAY_API_URL`, `AIRPAY_LOGIN`, `AIRPAY_PASSWORD`, таймаут и параметры повторов в окружении; для тестового и боевого стендов — разные значения.
3. Сохранять транзакцию до вызова `check`, а перед `pay` блокировать её от параллельного запуска.
4. При временном результате ставить операцию в очередь повторов и посылать в Airpay неизменный payload.
5. Строить интерфейс каталога динамически по `inputs`, `regexp`, `displays`, `contracts` и `invoice`.
6. Не логировать `Authorization`, `identityCard`, `fullName`, `account` и полные тела платёжных запросов.

## Наблюдения тестового контура Gamesales

Проверено 23 июля 2026 года:

- договор тестового агента ведётся в `RUB`; начальные `balance` и `overdraft` равны нулю;
- в каталоге доступны `A0001`, `A0002`, `A0003` (Казахстан) и `A0023` (международная тестовая услуга);
- `check` для `A0001` и `A0023` с `amountTo`/`amountFrom: 200.00` вернул `result: 204` — приём платежа в указанной валюте невозможен. Причину несовместимости суммы/валюты без контракта услуги определить нельзя;
- `check` для `A0023` без сумм вернул `result: 220` — недостаток средств у агента. Это подтверждает, что Airpay проверяет баланс уже на этапе `check`; при нулевых `balance` и `overdraft` нельзя получить успешный результат проверки;
- после повторной проверки `A0023` получен успешный `check`: Airpay вернул курс `RUB/KZT 5.9600`, `finalAmount: 5.9600 KZT`, а также данные плательщика в `displays`. Создан `transactionId`, но `pay` ещё не вызывался — списания не было;
- `currencyRate` и `finalAmount` описывают конвертацию и сумму зачисления провайдеру. Контракт всё ещё не содержит отдельного поля с комиссией или суммой списания агента; её можно подтвердить только сравнением `balance` до и после разрешённого тестового `pay`.

### Подтверждённое списание

Первоначальный вывод о списании `6.96 RUB` за платёж `A0023` с `amountTo: 1.00` и `amountFrom: 1.00` **не подтверждён**. До проверки баланса была проведена ещё одна успешная оплата `A0001`, поэтому конечный баланс нельзя отнести только к одной операции.

Подтверждённые данные более ранней оплаты `A0001`:

- `amountTo: 5.96 RUB`, `amountFrom: 6.96 RUB`;
- курс `RUB/KZT 5.9597435436`;
- `finalAmount: 35.52 KZT`;
- `amountTo × rate ≈ finalAmount`, а разница `amountFrom − amountTo` равна `1.00 RUB`.

Это показывает конвертацию и комиссию/надбавку для клиента, но не доказывает сумму дебета агентского баланса. Чтобы её измерить, нужно снять `balance` непосредственно до одной новой оплаты, выполнить только эту оплату и сразу повторить `balance`.

### Изолированная проверка дебета баланса

Для следующей отдельной операции `A0023` с `amountTo: 1.00` и `amountFrom: 1.00`:

- баланс непосредственно до `pay`: `-6.96 RUB`;
- баланс сразу после успешного `pay`: `-7.96 RUB`;
- фактический дебет: `1.00 RUB`.

Вывод, подтверждённый тестом: в проверенном сценарии Airpay списывает с баланса агента сумму `amountFrom`. `amountTo` задаёт сумму самой оплаты, а разница `amountFrom − amountTo` — сумма, добавленная к платежу клиента. `finalAmount` — результат конвертации в валюте провайдера и не является суммой дебета в валюте договора агента.
